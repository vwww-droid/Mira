# 我为什么做 Mira: 让 AI 从移动 App 沙盒视角理解风险环境

Mira 的起点不是一个完整的企业风控平台规划, 而是一个很具体的问题: 当一个风险环境看起来不对劲, 但我暂时找不到明确特征时, 能不能让 AI(人工智能)进入一个受系统权限限制的移动端视角, 帮我一起观察, 理解和沉淀经验。

在移动风控和安全开发里, 很多问题并不是某一个特征能决定的。模拟器, 云手机, 改机, Hook(运行时劫持), 无障碍滥用, 系统特征伪装和运行时注入经常交叉出现。真正有价值的也不是某一条检测代码, 而是这些现象为什么不对劲, 它们来自哪个权限边界, 还能用什么路径继续追。

Mira 想验证的关键想法很简单: AI 应该可以通过 MCP(Model Context Protocol, 模型上下文协议)进入 Android(安卓系统) 和 iOS(苹果移动系统) 的 App 沙盒视角, 读取团队沉淀的 Skill(技能), 观察所有可能不对劲的行为, 并把一次排查沉淀成可复查的 Case(案例)。

## 它不是完整平台

Mira 当前是一个 research prototype(研究原型)和 reference implementation(参考实现)。现在仓库里的 Android 和 iOS App 是 demo app(演示应用), 用来展示这套思路如何落地, 不是要求企业直接把这个 App 当作线上风控产品使用。

更自然的长期形态是 Android library(Android 库), iOS library(iOS 库)或 SDK(软件开发工具包)。企业可以把同样的端侧沙盒观察能力接入自己的业务 App, 再对接自己的情报系统, 工单系统, 服务端策略, 代码仓库, 多模型 review(审查), CI/CD(持续集成和持续交付), 灰度发布和回滚。

这些企业内部系统不应该由 Mira 重造。Mira 只聚焦其中最关键的一段: 端侧沙盒观察, MCP 接入, Skill 分析和本地 Case 沉淀。

## 我想象中的企业使用方式

假设业务侧发现了黑灰产线索, 例如某个私域售卖接口, 外挂能力说明, 群内售卖话术, 或某个被标记的账号和设备。当这个风险实体再次在线或触发业务行为时, 企业可以启动 AI 分析流程。

AI 先读取情报工单和业务上下文, 再通过 Mira MCP 连接业务 App 的沙盒会话。它不会突破 Android 或 iOS 的权限模型, 只在业务 App 自己能合法看到的范围里观察现象。

如果企业已经沉淀了黑灰产 Skill, AI 就可以按这些 Skill 去看注入痕迹, 无障碍状态, 系统特征异常, Hook 迹象, 模拟器或云手机线索。它的输出不应该是一个简单分数, 而是一组观察: 看到了什么, 来自哪里, 为什么不对劲, 还有哪些可能解释, 下一步要看什么。

如果没有发现可解释异常, 也不是失败。系统可以生成一个 Case, 写明已经观察了哪些范围, 哪些情报还没有被解释, 然后转给人继续看。

如果发现了新线索, 企业可以进一步结合自己的代码仓库, 修改风险标签, 上报字段, 策略解释或测试样例。之后让多个模型分别从误伤, 绕过, 性能, 隐私边界和回滚风险等角度 review。通过后再进入 Pull Request(代码合并请求), 人工审核, CI/CD, 灰度放量和回滚流程。

这条完整链路很大, 但 Mira 只做其中最关键的证明: AI 能不能从移动 App 的系统沙盒视角里理解风险环境。

## 为什么不是黑盒检测器

我不想把 Mira 做成一堆固定检测脚本。检测代码会过期, 业务场景也各不相同。真正值得沉淀的是判断方法: 什么现象值得说出来, 它来自哪个视角, 它为什么可能不对劲, 它还可能是什么正常解释。

所以 Mira 默认不提供黑盒检测代码, 而是维护 mobile risk skill(移动风险技能)和 case 报告。可疑行为有就是有, 先说出来, 再标注来源, 边界和下一步检查。

在 Android 上, 当前可以优先看当前进程和 shell(命令解释器) 可见范围, 例如 `/proc/self/maps`, `/proc/self/status`, `/proc/self/mountinfo`, `/proc/self/fd` 和系统公开接口。

在 iOS 上, `/mira` 是 Mira 提供的 app-view root(应用视角根目录), `/mira/proc` 是 simulated process view(模拟进程视图)。它们帮助 AI 理解 App 沙盒内的可见状态, 但不伪装成系统真实根目录或内核 procfs(进程文件系统)。

## 公开演示要有边界

这个项目确实会碰到黑灰产, 风控和逆向相关话题, 所以公开发布时必须收敛边界。

Mira 的公开演示只使用 synthetic case(合成案例), 自有设备或明确授权的环境。不要公开真实用户标识, 设备标识, 私有情报, 生产风控规则, token(令牌), tunnel(隧道)地址, 企业后台截图或第三方 App 数据。公网 Relay(中继)只适合临时演示或自托管测试, 演示结束后应该关闭。

这不是为了降低项目价值, 而是为了让研究保持可复现, 可审查和低攻击面。Mira 关注的是方法和参考实现, 不是公开真实对抗情报或生产策略。

## 接下来会怎么迭代

我会继续根据 Android 风险环境检测相关资料, 论文, 开源项目和真实排查案例, 提升 Mira 的基础观察能力。重点不是照搬检测代码, 而是理解每个检测点背后的系统原理, 可见边界和被对抗方式。

后续我会把更多高层 API(Application Programming Interface, 应用程序编程接口)观察逐步补充到 native bridge(原生桥接层), syscall(系统调用)包装和更贴近运行时底层的观察点。Android 侧也会探索带 inline assembly(内联汇编)的自有 native probe(原生探针)组件, 用来交叉验证高层 API 返回值。

更长期看, demo app 里的关键能力会逐步抽象成 Android library, iOS library 或 SDK。这样企业和研究者可以把这套思路接入自己的 App 和工程体系, 而不是围绕 Mira 的 demo app 重新搭平台。

Mira 首先是一个学习和研究项目。思路比实现更重要, 但一个能跑的最小实现, 能让思路被验证, 被讨论, 被改进。
